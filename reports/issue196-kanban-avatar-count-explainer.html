Kanban detail view counted avatar/delegate entries as Daemons

LingTai issue #196 · fix fix(tui): label kanban delegate count as avatars

One-line summary: The Ctrl+D detail view in /kanban read delegates/ledger.jsonl — which is the avatar/delegate spawn ledger — and counted every non-empty line as a "Daemon". The section is now labelled Avatars and counts only genuine avatar spawn records.

The bug before

In the agent properties view, pressing Ctrl+D opens a per-agent detail breakdown. One section reported a Daemons count. But the data source it read, delegates/ledger.jsonl, is the avatar topology ledger used elsewhere in the same codebase by fs.ReadLedger to build the network graph. Its records look like: 

{"event":"avatar","name":"gemini-fixer","working_dir":"gemini-fixer","ts":1718000000}

The old counting code in tui/internal/tui/props.go counted every non-empty line:

// Daemon spawn count from delegates/ledger.jsonl.
m.detailDaemonCount = 0
if data, err := os.ReadFile(filepath.Join(m.selectedDir, "delegates", "ledger.jsonl")); err == nil {
    for _, line := range strings.Split(string(data), "\n") {
        if strings.TrimSpace(line) != "" {
            m.detailDaemonCount++   // counts ANY line, incl. non-avatar events
        }
    }
}

Two problems compound: the label was wrong (avatar data shown as daemons), and the count was loose (any non-empty line, not just event == "avatar" records).

Impact

  • An agent with many avatars and zero daemons appeared to have many daemons.
  • The Kanban detail view misrepresented network topology and work distribution.
  • Any future non-avatar event line in the ledger would have inflated the number further.

The fix after

The smallest correct change: stop pretending this is daemon data. Relabel the section to Avatars and reuse the canonical ledger reader so the count matches the rest of the app's avatar topology — only event == "avatar" records are counted. 

// Avatar spawn count from delegates/ledger.jsonl. This ledger is the
// avatar/delegate spawn ledger (records with event == "avatar"), not a
// daemon run ledger — reuse the canonical reader so the count matches
// the network topology (see issue #196).
avatarEdges, _ := fs.ReadLedger(m.selectedDir)
m.detailAvatarCount = len(avatarEdges)

fs.ReadLedger already filters to event == "avatar" records with a non-empty working_dir, returning one edge per spawned avatar. Using len(edges) gives an exact, schema-aware count and removes a duplicate hand-rolled parser (the now-unused os import was dropped).

Terminology note: in LingTai, avatar = 分身 (full independent sub-agents) and daemon = 神识 (short-lived parallel workers). The ledger records avatars, so the Chinese label moves from 神识 to 分身 to match.

Tests

New regression tests in props_detail_test.go drive loadDetail() against a fixture ledger:

  • TestLoadDetailCountsOnlyAvatarSpawns — a ledger with two avatar records interleaved with a heartbeat event and a blank line yields a count of 2 (the loose line-counter would have returned 3).
  • TestLoadDetailNoLedger — a missing ledger yields 0 with no error.
